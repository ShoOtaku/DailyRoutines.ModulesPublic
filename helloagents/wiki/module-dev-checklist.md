# 模块开发 Checklist

> 适用范围: 在本仓库新增模块、重构模块、迁移模块或为模块增加较大功能时使用。  
> 使用方式: 按顺序逐项勾选，未完成项不得标记为完成。

## 1. 需求与范围确认
- [ ] 1.1 明确模块目标、输入、输出与边界（做什么/不做什么）
- [ ] 1.2 确认所属目录（`Action/`、`System/`、`UIOptimization/` 等）并避免跨目录职责重叠
- [ ] 1.3 确认是否涉及高风险行为（EHRB），若涉及先做风险规避与回退设计

## 2. 设计与拆分
- [ ] 2.1 将功能拆分为可独立验证的小任务（建议每项可单独构建验证）
- [ ] 2.2 明确依赖项（主插件 API、第三方程序集、配置项、事件源）
- [ ] 2.3 明确失败路径与降级策略（异常、超时、依赖缺失、UI 不可用）

## 3. 代码实施
- [ ] 3.1 新建或修改对应 `.cs` 文件，保证命名与目录语义一致
- [ ] 3.2 遵循 `.editorconfig` 与 `wiki/coding-style.md` 规范
- [ ] 3.3 避免引入无关重构；单次提交聚焦单一主题
- [ ] 3.4 对关键逻辑添加最少必要注释（解释“为什么”，不是复述代码）

## 4. 静态检查与构建
- [ ] 4.1 本地构建通过：`dotnet build DailyRoutines.ModulesPublic.csproj -c Debug`
- [ ] 4.2 检查 `DailyRoutines.CodeAnalysis` 结果，不引入新的高严重度问题
- [ ] 4.3 若需新增 `NoWarn`，已记录必要性、影响范围与后续清理计划

## 5. 运行验证（最小可验证）
- [ ] 5.1 覆盖至少 1 个主流程场景（预期成功）
- [ ] 5.2 覆盖至少 1 个异常/边界场景（预期安全失败）
- [ ] 5.3 验证不影响相邻模块关键路径（无明显回归）

## 6. 知识库同步
- [ ] 6.1 更新对应模块文档：`helloagents/wiki/modules/<module>.md`
- [ ] 6.2 若涉及通用规范，更新 `helloagents/wiki/coding-style.md`
- [ ] 6.3 在 `helloagents/CHANGELOG.md` 记录本次重要变更
- [ ] 6.4 若使用方案包流程，确保 `plan/` 与 `history/` 状态一致

## 7. 提交前最终核对
- [ ] 7.1 变更文件清单可解释，每个文件改动有明确目的
- [ ] 7.2 已清理临时代码/调试开关/无用引用
- [ ] 7.3 提交信息准确描述“做了什么 + 为什么”
- [ ] 7.4 可回滚性明确（删除/回退策略清晰）
